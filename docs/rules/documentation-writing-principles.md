# md-to-lark 文档写作原则

本文用于约束 `docs/` 下的正式文档。

这些文档面对的不是单一读者，而是三类逐步深入的使用者：

1. 第一次接触项目、只想成功跑起来的人。
2. 已经在使用工具、需要掌握复杂配置和高级能力的人。
3. 需要调试、扩展或二次集成、想理解内部实现的人。

目标不是把所有技术细节都堆进每一篇文档，而是让读者按正确顺序获得三种能力：

1. 知道这工具解决什么问题，以及最短使用路径是什么。
2. 知道遇到真实复杂输入时，应该如何配置、观察和排查。
3. 知道这套工具内部如何工作，出了问题从哪里查起。

## 1. 适用范围

适用于以下类型的文档：

1. 入门文档。
2. 使用指南。
3. 参数与接口参考文档。
4. 架构与实现文档。
5. 调试与排障文档。

不适用于以下类型的文档：

1. 迁移计划文档。
2. 临时讨论纪要。
3. 一次性验收记录。
4. 发布过程备忘录。

## 1.1 规则文档自身的维护原则

这份规则文档本身也是文档。每次修改时，不能只改局部句子，必须重新通读全文，再判断它是否仍然合理。

每次修改前，至少重新检查三件事：

1. 规则分类是否仍然清楚。
2. 规则详细度是否仍然合适。
3. 规则是否仍然配有足够例子。

## 2. 核心写作原则

### 2.1 先讲用户问题，再讲功能，再讲实现

每篇文档开头都先回答：

1. 这篇文档要解决什么问题。
2. 读者什么时候应该来看这篇。
3. 看完之后能完成什么动作，或获得什么判断能力。

不要一上来从源码目录、类名、函数名或模块名写起。

### 2.2 每篇文档必须属于一个层级

`docs/` 里的文档只能属于以下四层之一：

1. `01-getting-started`
2. `02-guides`
3. `03-reference`
4. `04-internals`

写作时先判断文档属于哪一层，再动笔。

不允许把“第一次怎么用”和“内部怎么实现”混在同一篇里。

### 2.3 一篇文档只回答一个核心问题

每篇文档只能围绕一个主问题展开。

例如：

1. 如何发布单个 Markdown 文件。
2. 如何使用 preset 改写 Markdown。
3. 为什么 `single H1` 会被提升为文档标题。
4. `LAST -> BTT -> Lark Docx` 是如何映射的。

不要在同一篇文档里同时讲：

1. 快速开始。
2. 全量 CLI 参数。
3. Mermaid 行为差异。
4. 内部渲染实现。

### 2.4 先给最短可执行路径，再解释原理

入门和使用指南类文档，必须优先给出可以直接执行的命令、配置或输入示例。

推荐顺序是：

1. 先给结论。
2. 再给最短操作步骤。
3. 再解释默认行为和关键限制。
4. 最后再补原因和实现映射。

### 2.5 默认读者知道命令行，但不知道项目约定

正文不要假设读者完全不懂技术，但也不要假设读者已经熟悉本项目内部术语。

第一次出现以下概念时，必须先用人话解释，再给术语名：

1. 预设转换（preset transform）
2. 预处理（prepare）
3. 中间表示（LAST / BTT）
4. 渲染到飞书文档（render to Feishu Docx）
5. dry-run
6. pipeline cache

第一次出现英文术语时，统一写成“中文（English）”。

例如：

1. 中间表示（Intermediate Representation）
2. 预设转换（Preset Transform）
3. 干跑（dry-run）

如果后文继续使用，优先使用中文；代码、命令、环境变量和路径里保留原始标识符。

### 2.6 必须使用真实输入和真实命令示例

示例优先使用仓库里真实存在的输入和脚本入口。

例如：

1. `npm run publish:md -- --input ./test-md/comp/comp.md --dry-run`
2. `npm run publish:md -- --input ./test-md/mermaid.md --mermaid-target board`
3. `npm run fetch:board-data -- --doc <document_id> --index 1`

不要写脱离项目上下文的假命令，也不要只写抽象描述。

### 2.7 必须明确区分默认行为、可选配置和边界条件

文档里凡是会影响结果的能力，都应区分成三层：

1. 默认会发生什么。
2. 用户可以怎样改。
3. 哪些情况会失败、退化或不生效。

例如：

1. 标题策略默认会加日期前缀，但可用 `--no-date-prefix` 关闭。
2. 本地缺失附件不会让整次发布失败，而是退化为文本。
3. `--doc` 只适用于单文件模式，不适用于目录模式。

### 2.8 每篇文档都要告诉读者下一步去哪

文档结尾必须至少有一个“相关文档”或“下一步阅读”段落。

推荐做法：

1. 入门文档指向下一篇 guide。
2. guide 指向相关 reference。
3. reference 指向 internals。
4. internals 指向源码入口。

### 2.9 只有 internals 文档才把源码作为主线

源码文件、模块边界和调用链只在 `04-internals` 里作为正文主线出现。

其他层级如果确实要提源码，只能放在文末的“实现映射”或“源码入口”小节。

### 2.10 参考文档只做查询，不做教学

`03-reference` 下的文档应尽量稳定、扁平、可查找。

它们适合承载：

1. CLI 参数表。
2. 环境变量表。
3. preset 模块导出约束。
4. 程序化调用的输入输出。

它们不适合承载：

1. 初学者教程。
2. 长篇背景说明。
3. 架构推导过程。

## 3. 推荐文档结构

### 3.1 入门文档模板

推荐结构：

1. 这篇文档解决什么问题
2. 什么时候看这篇
3. 最短步骤
4. 你会看到什么结果
5. 常见失败
6. 下一步阅读

### 3.2 使用指南模板

推荐结构：

1. 这个能力解决什么问题
2. 默认行为
3. 常用配置
4. 真实示例
5. 边界条件与失败模式
6. 相关文档

### 3.3 参考文档模板

推荐结构：

1. 这份参考覆盖什么范围
2. 参数或字段表
3. 默认值
4. 限制条件
5. 相关 guide / internals

### 3.4 实现文档模板

推荐结构：

1. 这部分内部机制解决什么问题
2. 总体调用链
3. 关键数据模型
4. 关键设计选择
5. 调试入口
6. 相关源码入口

## 4. 写作检查清单

每次写完一篇文档，至少检查下面这些问题：

1. 这篇文档属于哪一层，是否和其他层内容混写了。
2. 开头是否先回答了读者问题，而不是直接讲源码。
3. 是否给了最短可执行示例。
4. 是否写清楚默认行为、可选配置和失败边界。
5. 是否使用了仓库里的真实命令、真实路径和真实输入。
6. 是否给出了下一步阅读路径。
7. 如果这是一篇 internals 文档，是否给了源码入口。

## 5. 反例

下面这些写法不符合本规则：

1. 在快速开始文档里直接展开 `LAST`、`BTT` 和 block 树细节。
2. 在 reference 文档里写成长篇教程。
3. 在 guide 文档里只列参数，不解释默认行为和失败边界。
4. 通篇只说“支持 Mermaid、支持附件”，但不给真实命令和真实输入。
5. 用 `src/commands/...`、`src/pipeline/...` 作为新手文档的小节标题。
