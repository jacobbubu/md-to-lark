# md-to-lark 是什么

## 这篇文档解决什么问题

这篇文档回答一个问题：`md-to-lark` 到底是什么工具，适合在什么场景下使用。

如果你还不知道这套工具和普通 Markdown 渲染器有什么区别，先看这篇。

## 什么时候看这篇

适合下面几类情况：

1. 你第一次接触这个仓库，想知道它解决什么问题。
2. 你已经知道它和飞书文档有关，但还不知道支持到什么程度。
3. 你想判断自己的输入场景是否适合直接用这套工具。

## 一句话结论

`md-to-lark` 是一套把 Markdown 稳定发布到飞书云文档的工具链。

它不是只做“把 Markdown 渲染出来”，而是把真实文档发布里常见的复杂输入一起处理掉，例如附件、远程图片、Mermaid、表格、iframe 类链接、标题策略和批量发布。

## 它解决什么问题

很多团队的内容原本是 Markdown，但真正交付的目标不是本地预览，而是飞书文档。

这时常见的问题不是“能不能转成文本”，而是下面这些更具体的事情：

1. 一个目录下的多篇 Markdown 怎么稳定批量发布。
2. 标题到底来自文件名、手工指定标题，还是 Markdown 里的单个 H1。
3. 本地图片和附件怎么上传并替换成飞书可识别的资源。
4. 远程图片和独立 URL 怎么在发布前预处理。
5. Mermaid 是渲染成文本图，还是渲染成飞书白板。
6. 表格怎么做列宽启发式、数字列右对齐和大表格扩容。
7. 发布失败时，怎么知道问题出在输入、预处理、中间表示，还是飞书 API。

`md-to-lark` 的价值就在于，它把这些现实问题放进同一条可重复执行的发布链路里。

## 它能处理哪些输入

基于当前代码和样例，这套工具已经覆盖了下面这些常见输入：

1. 单个 Markdown 文件，或整个目录下递归的 `*.md` 文件。
2. 常见富文本样式，例如粗体、斜体、删除线、行内代码、引用、列表和 todo。
3. 本地图片和本地附件。
4. 远程 Markdown 图片链接。
5. Mermaid 代码块。
6. KaTeX 行内公式和块级公式。
7. 带有 iframe 语义的链接，例如嵌入型链接。
8. 复杂表格，包括数字列对齐和大表格扩展。

样例输入可以直接看：

1. [`test-md/comp/comp.md`](/Users/rongshen/vibe-coding/new/md-to-lark/test-md/comp/comp.md)
2. [`test-md/mermaid.md`](/Users/rongshen/vibe-coding/new/md-to-lark/test-md/mermaid.md)

## 它是怎么工作的

从使用者视角看，这条链路可以理解成五步：

1. 读取一个 Markdown 文件，或解析一个目录里的所有 Markdown。
2. 如果配置了预设转换（Preset Transform），先改写 Markdown。
3. 做预处理，例如下载远程图片、提取独立 URL 媒体、重写链接。
4. 把 Markdown 逐步转换成内部中间表示，再转换成飞书块树。
5. 根据模式选择：
   直接输出 dry-run 结果，或者真正写入飞书文档。

如果你后面要理解实现细节，可以继续看 internals 文档；第一次使用时，只需要知道这不是“单步转换器”，而是一条带中间产物和排障信息的发布流水线。

## 一个真实例子

第一次了解这套工具时，最直观的方式是跑一次真实样例的 dry-run：

```bash
npm run publish:md -- --input ./test-md/comp/comp.md --dry-run
```

这条命令不会写飞书文档，但会完整走一遍发布流水线，并把中间产物写到默认的 pipeline cache 目录。

注意，当前 CLI 即使在 `--dry-run` 模式下，也仍然会先校验飞书相关环境变量，所以你仍然需要准备好最小的 `.env` 配置。

## 什么时候不该先看实现

如果你的目标只是：

1. 先把一篇 Markdown 成功跑通。
2. 先确认 dry-run 会产出什么。
3. 先知道环境变量该怎么配。

那就不要先看架构文档，直接继续看 quickstart。
